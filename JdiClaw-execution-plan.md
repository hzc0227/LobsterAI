# JdiClaw 改造主计划

## 目标

本计划用于把当前 `LobsterAI` 以“最小改造、最低风险、最少外部依赖”的方式逐步改造成 `JdiClaw`。

本次计划不再按“功能完整度优先”推进，而改为按以下顺序推进：
- 先做低复杂度、低风险、可直接验收的品牌与壳层改造
- 再做中复杂度的技术收口，为后续接 JD 能力降低改动面
- 最后才做高复杂度、高外部依赖的登录、模型、技能、发布与迁移

## 核心策略

### 策略一：最小改造优先

- 优先改“用户能直接看到”的部分
- 延后改“系统身份和运行时强绑定”的部分
- 延后改“需要 JD 后端或平台配合”的部分
- 延后改“会影响老数据、协议、安装包身份”的部分

### 策略二：复杂度优先

- 复杂度 `1/5`：几乎无外部依赖，适合直接交给 AI 执行
- 复杂度 `2/5`：需要少量代码上下文，但风险仍然可控
- 复杂度 `3/5`：需要跨模块收口，必须限制改动范围
- 复杂度 `4/5`：需要服务端、网关或鉴权配合
- 复杂度 `5/5`：涉及身份迁移、打包发布、数据库、协议或企业治理

### 策略三：单任务可验收

后续给 AI 的任务必须满足：
- 一次只做一个明确点
- 最好只改 `1-6` 个文件
- 必须明确允许修改的文件范围
- 必须明确禁止触碰的范围
- 必须有停止条件
- 必须有完成标准与验证方式

## 不再采用的旧推进方式

以下做法不再作为默认路线：
- 一上来就改 JD 登录
- 一上来就改服务端模型和网关
- 一上来就改 `APP_ID`、deep link、数据库文件名
- 在品牌还没稳定前提前做迁移、打包、更新、权限治理

## 当前已知约束

- 当前项目的主进程入口在 `src/main/main.ts`
- 应用基础常量在 `src/main/appConstants.ts`
- 服务端登录、用户信息、模型列表相关逻辑集中在 `src/main/main.ts`
- 服务端基址在 `src/main/libs/endpoints.ts`
- 服务端模型 fallback 与代理链路在 `src/main/libs/claudeSettings.ts`
- 渲染层模型配置在 `src/renderer/config.ts` 与 `src/renderer/services/config.ts`
- 登录态与服务端模型加载在 `src/renderer/services/auth.ts`
- Cowork UI 主流程在 `src/renderer/components/cowork/`
- 自动更新、登录地址下发、技能商店仍绑定有道系地址，位于 `src/renderer/services/endpoints.ts`

## 总体阶段

| 阶段 | 名称 | 复杂度 | 外部依赖 | 目标 |
| --- | --- | --- | --- | --- |
| Phase 0 | 改造规则冻结 | 1/5 | 无 | 定义边界与替换顺序 |
| Phase 1 | 品牌表层替换 | 1/5 | 无 | 先把项目“看起来像 JdiClaw” |
| Phase 2 | 品牌壳层替换 | 2/5 | 无 | 扩展到窗口、菜单、设置、资源壳层 |
| Phase 3 | 技术接缝收口 | 2-3/5 | 低 | 为后续 JD 接入降低改动面 |
| Phase 4 | 外部能力接入 | 4-5/5 | 高 | 最后接 JD 登录、模型、技能、内网 API |
| Phase 5 | 发布与身份迁移 | 4-5/5 | 高 | 最后处理安装包身份、协议、迁移、更新 |

## Phase 0：改造规则冻结

### 目标

在真正推进改造前，先冻结“先改什么、后改什么、什么暂时不动”。

### 任务

- [ ] 明确本轮改造优先级：品牌优先，登录/API/技能后置。
- [ ] 明确本轮“最小改造”边界：先不改 `APP_ID`、deep link、数据库文件名、自动更新、技能执行逻辑。
- [ ] 梳理需要后置的高风险项：JD 登录、模型网关、服务端模型、内网 API、技能、迁移、打包签名。
- [ ] 梳理可前置的低风险项：logo、名称、README、窗口标题、菜单文案、设置页展示文案。
- [ ] 为后续 AI 执行准备统一任务模板与拆分任务目录。

### 完成标准

- 主计划与任务模板已经写好
- 可以用统一格式持续拆分任务
- 新任务不会再反复讨论优先级

## Phase 1：品牌表层替换

### 目标

先完成所有“直接面向用户、低风险、低耦合”的品牌替换，让项目视觉与文案先变成 `JdiClaw`。

### 复杂度

- 复杂度：`1/5`
- 风险：低
- 适合直接派给 AI 独立执行

### 范围

#### In Scope

- README / README_zh / 项目简介
- Logo 与图标引用
- 渲染层可见产品名
- 设置页、帮助页、说明页的品牌文案

#### Out of Scope

- 登录逻辑
- API 调用
- provider 与模型请求目标
- deep link、数据库、安装包身份

### 任务包

- `BRAND-01` 文档品牌替换
- `BRAND-02` Logo 与资源引用替换
- `BRAND-03` 渲染层可见品牌文案替换

### 完成标准

- README 与界面上的主要产品名已经切换为 `JdiClaw`
- 用户肉眼可见的品牌元素已基本替换
- 构建不受影响

## Phase 2：品牌壳层替换

### 目标

在表层品牌替换完成后，再扩展到桌面应用壳层，包括窗口、托盘、菜单、帮助入口等。

### 复杂度

- 复杂度：`2/5`
- 风险：低到中
- 仍适合单任务派发，但要控制改动文件数

### 范围

#### In Scope

- 窗口标题
- 托盘与菜单文案
- About / 帮助入口文案
- 部分资源文件显示名

#### Out of Scope

- `APP_ID`
- deep link 协议
- 数据库文件名
- 更新源与服务端地址

### 任务包

- `BRAND-04` 主进程窗口/菜单/托盘品牌替换
- `BRAND-05` 壳层文案与帮助入口品牌替换

### 完成标准

- 桌面应用运行时的主要壳层品牌已切换
- 不影响现有登录、模型与 Cowork 能力

## Phase 3：技术接缝收口

### 目标

不直接接 JD 能力，而是先把散落的品牌名、endpoint 文案、provider 外显名、帮助入口等收口，降低后续高风险改造的代码接触面。

### 复杂度

- 复杂度：`2-3/5`
- 风险：中
- 适合白天空闲时执行，执行前要限定修改边界

### 范围

#### In Scope

- 品牌常量收口
- 显示层字符串收口
- endpoint 常量与外显命名收口
- 为后续 JD 接入预留统一配置位

#### Out of Scope

- 真实切换到 JD 登录
- 真实切换到 JD 模型网关
- 真实替换技能执行链路

### 任务包

- `SAFE-01` 品牌常量收口
- `SAFE-02` endpoint 与外显命名收口
- `SAFE-03` 为未来 JD 接入预留配置入口

### 完成标准

- 品牌与服务端字符串不再散落在大量文件中
- 后续 Phase 4 改造时不需要大面积全文替换

## Phase 4：外部能力接入

### 目标

在品牌与技术接缝稳定后，最后再接入 JD 的外部能力，包括登录、鉴权、模型网关、服务端模型、技能与内网 API。

### 复杂度

- 复杂度：`4-5/5`
- 风险：高
- 不建议在信息不完整时让 AI 自行猜测实现

### 范围

#### In Scope

- JD 登录链路
- `code -> token` / `refresh token`
- 用户信息接口
- 服务端模型列表
- 模型网关与代理
- JD 技能与内网 API

#### Out of Scope

- 仍然不立即处理安装包身份迁移
- 仍然不立即处理数据库名迁移

### 任务包

- `AUTH-01` JD 登录入口与回调链路
- `AUTH-02` token 存储与刷新逻辑
- `MODEL-01` 服务端模型列表与 fallback 改造
- `MODEL-02` 模型网关与 Cowork 最小链路打通
- `SKILL-01` JD 技能与内网 API 接入

### 前置依赖

- JD 登录页地址
- deep link 回调协议
- token 交换与刷新接口
- 用户信息接口
- 模型列表接口
- 模型网关或代理接口

### 完成标准

- 至少一条 JD 登录链路可用
- 至少一条 JD 模型调用链路可用
- Cowork 可以跑最小任务

## Phase 5：发布与身份迁移

### 目标

把所有真正影响安装身份、运行身份和历史兼容的改动放到最后处理。

### 复杂度

- 复杂度：`4-5/5`
- 风险：高
- 必须在前面阶段稳定后再做

### 范围

#### In Scope

- `APP_ID`
- deep link 协议
- 数据库文件名
- 打包显示名
- 自动更新
- 安装包签名
- 历史数据迁移

#### Out of Scope

- 无关当前发布与迁移的 UI 微调

### 任务包

- `RELEASE-01` 打包与安装器身份替换
- `RELEASE-02` 自动更新链路切换
- `MIGRATION-01` deep link 与数据库身份迁移
- `MIGRATION-02` 历史数据兼容与回滚策略

### 完成标准

- 可以生成 JdiClaw 安装包
- 能安装、升级、迁移
- 历史 LobsterAI 数据处理路径明确

## 当前推荐执行顺序

1. 先做 `Phase 0`
2. 连续完成 `Phase 1`
3. 再做 `Phase 2`
4. 视精力做 `Phase 3`
5. 等 JD 外部条件准备好后，再做 `Phase 4`
6. 所有高风险身份改动统一放到 `Phase 5`

## 任务派发规则

后续每次给 AI 派任务时，必须满足以下约束：

- 只派发一个任务文件
- 明确允许修改哪些文件
- 明确禁止触碰哪些文件
- 明确停止条件
- 明确完成标准
- 明确验证命令和人工验收点

## 最适合优先执行的任务类型

### 最适合现在先做

- `BRAND-*`
- `SAFE-*`

### 建议等信息明确后再做

- `AUTH-*`
- `MODEL-*`
- `SKILL-*`
- `RELEASE-*`
- `MIGRATION-*`

## 完成定义

### 这轮“最小改造”完成定义

满足以下条件即可认为完成一轮最小改造：
- 品牌表层和壳层已切换为 `JdiClaw`
- 品牌与服务端相关字符串已做统一收口
- JD 登录/API/技能尚未接入也不会阻碍后续继续改造
- 项目仍可正常构建与运行

## 备注

本主计划默认你当前的真实诉求是：

- 先做最小改造
- 先做品牌与 logo
- 把 JD 登录、API、技能放到最后
- 把任务拆到足够细，方便随时交给 AI 执行

后续如果策略再变，例如要提前切登录或要提前打包，再在本文件上调整阶段优先级，不要直接越阶段执行。
